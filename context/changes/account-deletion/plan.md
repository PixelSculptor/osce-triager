# Account Deletion with 30-Day RODO Retention — Implementation Plan

## Overview

Implement S-05: a student can request account deletion from a new `/account/settings` page. The request sets `deletionRequestedAt = NOW()` on the `user` row; the account enters a 30-day soft-delete window during which the student can still log in and cancel. A GitHub Actions nightly workflow hard-deletes accounts whose `deletionRequestedAt` is older than 30 days — triggering existing `ON DELETE CASCADE` constraints that purge all related rows automatically. No Cloudflare Worker changes needed.

## Current State Analysis

- `users` table (`src/shared/lib/schema.ts:11-20`): 6 columns, no `deletionRequestedAt` field.
- All FK relations (`account`, `session`, `session_result`) already use `ON DELETE CASCADE` — hard delete cascades automatically to all child rows.
- `authorize` callback (`src/modules/auth/auth.ts:39-56`): no deletion-state guard. No changes needed for the cancellable flow (users stay able to log in during the 30-day window).
- No `/account/settings` route exists.
- `Nav.tsx` (`src/shared/components/Nav/Nav.tsx`): shows email + logout button for authenticated users; no Settings link.
- `DATABASE_URL` is already a GitHub Secret (`deploy.yml:24-25`) — the cleanup workflow reuses it with no new provisioning.
- JWT strategy: calling nothing on deletion request (user stays logged in). `signOut()` is not needed because the user must be able to log in and cancel.

## Desired End State

1. Student navigates to `/account/settings` via a Nav link (visible when authenticated).
2. Settings page shows either the delete form (no pending request) or the "pending deletion" banner (request active).
3. Student types `DELETE` in the confirmation input → submit → `deletionRequestedAt` set → page shows pending state with a "Cancel deletion" button.
4. Student can cancel at any time within 30 days: click "Cancel deletion" → `deletionRequestedAt = NULL` → page shows delete form again.
5. After 30 days: nightly GitHub Actions job hard-deletes the user row; cascade removes `account`, `session`, `session_result`, `session_event` rows. Student can no longer log in (user row gone → `authorize` returns `null`).

### Key Findings

- `deletionRequestedAt timestamp("deletion_requested_at", { mode: "date" })` follows the exact camelCase-TS / snake_case-DB naming used throughout the schema (e.g., `hashedPassword: text("hashed_password")` at `schema.ts:19`).
- New migration 0002 needed — generated by `drizzle-kit generate`, applied by `drizzle-kit migrate`.
- Query module pattern required by `lessons.md` — settings page must import from `src/modules/account/queries.ts`, not do DB calls inline.
- `requestDeletionAction` and `cancelDeletionAction` go in a new `src/modules/account/actions.ts` — separate from `src/modules/auth/actions.ts` to keep auth and account lifecycle concerns apart.
- Cleanup script uses `postgres` (already a dep) with `{ prepare: false }` — the Supabase Transaction Pooler constraint from `auth-scaffold/plan.md:39`.

## What We Are NOT Doing

- Signing the user out on deletion request — they must stay logged in to cancel from settings.
- Blocking login for pending-deletion accounts — the 30-day window is an active grace period.
- Adding a Cloudflare Cron Trigger — `DATABASE_URL` would need a new Worker secret; GitHub Actions reuses the existing secret.
- Sending a confirmation email — no email provider; request is logged server-side only.
- Changing any `ON DELETE CASCADE` constraints — they are correct for the hard-delete step.
- Adding WCAG or i18n polish — out of scope per PRD.

## Implementation Approach

Three sequential phases, each independently verifiable:

1. **Schema** — add `deletionRequestedAt` column and apply migration. No other code changes yet; safe to merge standalone.
2. **UI + Actions** — settings page, two server actions, Nav link. Relies on Phase 1 column existing in production schema.
3. **Cleanup workflow** — GitHub Actions nightly job. Relies on Phase 1 column to query against. Can be merged independently of Phase 2.

## Critical Implementation Details

**`requestDeletionAction` must validate the confirmation text server-side.** The client disables the submit button when input ≠ `"DELETE"`, but the server action must also check `formData.get("confirmation") !== "DELETE"` and return an error — client-side guard alone is insufficient.

**`db.query.users.findFirst` in `authorize` already handles hard-deleted users.** When the nightly job deletes a user row, `findFirst` returns `undefined` → `authorize` returns `null` → login rejected. No extra guard needed after hard delete.

## Phase 2: Settings Page and Server Actions

### Overview

New `/account/settings` route with a "Delete account" section (type-DELETE modal) and "Cancel deletion" banner. Two server actions manage the state. A Nav link links to the page.

### Required Changes

#### 1. Account query module

**File**: `src/modules/account/queries.ts` *(new)*

**Purpose**: Server-only query that returns the user's account settings, including `deletionRequestedAt`, for the settings page RSC. Follows the query-module pattern from `lessons.md`.

**Contract**: Export `getAccountSettings(userId: string)` returning a single user row with fields `id`, `email`, `name`, `deletionRequestedAt`. Use `db.query.users.findFirst` with a `columns` projection. Add `import "server-only"` at top.

#### 2. Account server actions

**File**: `src/modules/account/actions.ts` *(new)*

**Purpose**: Two server actions that mutate `deletionRequestedAt`. Both must start with `"use server"`, call `auth()` for the userId, and return an action state type.

**Contract**:

`requestDeletionAction(prevState, formData: FormData): Promise<AccountActionState>` —
  1. Call `auth()` → if no session, return `{ error: "Unauthorized" }`.
  2. Read `formData.get("confirmation")` → if not `"DELETE"`, return `{ error: "Wpisz DELETE aby potwierdzić" }`.
  3. `db.update(users).set({ deletionRequestedAt: new Date() }).where(eq(users.id, session.user.id))`.
  4. Return `{ success: true }`.

`cancelDeletionAction(): Promise<void>` —
  1. Call `auth()` → if no session, return silently.
  2. `db.update(users).set({ deletionRequestedAt: null }).where(eq(users.id, session.user.id))`.
  3. Return (no redirect — page re-renders via revalidation).

Both actions call `revalidatePath("/account/settings")` after the DB update so the page re-fetches fresh data.

**AccountActionState type** (define at top of file):
```ts
export type AccountActionState = { error?: string; success?: boolean } | null
```

#### 3. Settings page (RSC)

**File**: `src/app/account/settings/page.tsx` *(new)*

**Purpose**: Server component that loads account data and renders the appropriate section. Follows the same auth-then-redirect pattern as `src/app/dashboard/page.tsx:7-8`.

**Contract**: Call `auth()` → if no session `redirect("/login")`. Call `getAccountSettings(session.user.id)`. If `user.deletionRequestedAt IS NOT NULL`, render `<CancelDeletionSection deletionDate={user.deletionRequestedAt} />`. Otherwise render `<DeleteAccountSection />`. Wrap in a `<main>` with a heading ("Ustawienia konta").

#### 4. Delete account client component

**File**: `src/app/account/settings/DeleteAccountSection.tsx` *(new)*

**Purpose**: `'use client'` component with a controlled text input; disables the submit button until the input equals `"DELETE"`. Uses `useActionState` with `requestDeletionAction`.

**Contract**: Import `requestDeletionAction` and `AccountActionState`. Use `useActionState(requestDeletionAction, null)`. Render a `<form action={formAction}>`: hidden or visible `<input name="confirmation">` where `value` is bound to local `useState`; a submit `<button disabled={inputValue !== "DELETE"}>`. Show `state?.error` if present. Show a success message if `state?.success`.

#### 5. Cancel deletion client component

**File**: `src/app/account/settings/CancelDeletionSection.tsx` *(new)*

**Purpose**: `'use client'` component shown when `deletionRequestedAt IS NOT NULL`. Shows deletion date and a "Cancel deletion" button.

**Contract**: Accept `deletionDate: Date` prop. Compute `purgeDate = deletionDate + 30 days` for display. Render a warning banner and a `<form>` with `formAction={cancelDeletionAction}`.

#### 6. Nav — Settings link

**File**: `src/shared/components/Nav/Nav.tsx`

**Purpose**: Add a "Ustawienia" link for authenticated users, following the existing link pattern.

**Contract**: Inside the `{session ? (<>…</>)}` branch at `Nav.tsx:17`, add `<Link href="/account/settings" className={styles.settingsLink}>Ustawienia</Link>` between the email `<span>` and the logout `<form>`. Add the `.settingsLink` class to `Nav.module.css` (same style as existing links).

### Criteria for Success

#### Automated Verification

- Typecheck passes: `npx tsc --noEmit`
- Lint passes: `npm run lint`

#### Manual Verification

- Authenticated user sees "Ustawienia" in Nav → clicking navigates to `/account/settings`.
- Settings page loads: shows "Usuń konto" section with an input and a disabled button.
- Typing anything other than `DELETE` → button stays disabled.
- Typing `DELETE` → button enabled; submitting → page updates to show "pending deletion" banner with purge date.
- Clicking "Anuluj usunięcie" → page reverts to the delete form.
- Unauthenticated user visiting `/account/settings` → redirected to `/login`.

## Phase 3: Scheduled Cleanup (GitHub Actions)

### Overview

Nightly GitHub Actions workflow that connects to the production database and hard-deletes users whose `deletionRequestedAt` is older than 30 days. The existing `ON DELETE CASCADE` constraints handle all child-row cleanup.

### Required Changes

#### 1. Cleanup script

**File**: `scripts/cleanup-expired-accounts.mjs` *(new)*

**Purpose**: Standalone Node.js ES module script; importable and runnable locally for verification. Keeps the logic versioned alongside the schema it targets.

**Contract**: Uses `postgres` (already a dep) with `{ prepare: false }` (Supabase PgBouncer constraint). Reads `DATABASE_URL` from `process.env`. Runs:
```sql
DELETE FROM "user"
WHERE deletion_requested_at IS NOT NULL
  AND deletion_requested_at < NOW() - INTERVAL '30 days'
```
Logs deleted count. Calls `sql.end()` before exit. Exits with code 1 on error.

#### 2. Cleanup workflow

**File**: `.github/workflows/cleanup.yml` *(new)*

**Purpose**: Scheduled GitHub Actions workflow that runs the cleanup script nightly with production credentials. Also has `workflow_dispatch` for on-demand manual runs (useful for testing).

**Contract**:

```yaml
name: 🧹 Nightly account deletion cleanup

on:
  schedule:
    - cron: '0 2 * * *'  # 02:00 UTC daily
  workflow_dispatch:

jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: Delete expired accounts
        run: node scripts/cleanup-expired-accounts.mjs
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

`DATABASE_URL` secret is already present in the repo from `deploy.yml`.

### Criteria for Success

#### Automated Verification

- Script runs without error against an empty/dev database: `DATABASE_URL=<dev-url> node scripts/cleanup-expired-accounts.mjs`
- TypeScript types consistent (script is `.mjs`, no TS compilation needed).

#### Manual Verification

- In GitHub Actions UI: manually trigger "Nightly account deletion cleanup" via `workflow_dispatch` → workflow completes successfully → logs show "Deleted 0 accounts" (or N if test data exists).
- Set a test user's `deletion_requested_at` to 31 days ago → trigger manually → user row gone from DB.

## Testing Strategy

### Automated

- `npx tsc --noEmit` must pass after each phase.
- `npm run lint` must pass after Phase 2.

### Manual Flows to Verify

1. **Happy path – delete**: Log in → Settings → type "DELETE" → confirm → see pending banner with purge date.
2. **Happy path – cancel**: After step 1, click "Cancel deletion" → see delete form again.
3. **Confirmation guard**: Type anything other than `"DELETE"` → button stays disabled; submit should not be possible.
4. **Unauthenticated access**: Visit `/account/settings` without a session → redirected to `/login`.
5. **Cleanup script**: Manually insert a user with `deletion_requested_at = NOW() - INTERVAL '31 days'` in dev DB → run script → user and all related rows gone.
6. **Post-hard-delete login attempt**: After step 5, attempt to log in with deleted user's credentials → rejected ("Nieprawidłowy email lub hasło").
